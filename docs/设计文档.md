```
┌──────────────────┐   HTTP/JSON    ┌──────────────────┐     gRPC      ┌──────────────────┐
│   Web UI          │ ←──────────→  │   API Server     │ ←──────────→ │   drop_server    │

│   :5000           │               │   :5000          │              │   :50051         │
└──────────────────┘               └──────────────────┘              └────────┬─────────┘
                                                                             │ gRPC
                                                                    ┌────────▼─────────┐
                                                                    │   drop_agent     │
                                                                    │   C++17          │
                                                                    │   fork+exec perf │
                                                                    └────────┬─────────┘
                                                                             │ perf.data
                                                                    ┌────────▼─────────┐
                                                                    │   Analyzer       │
                                                                    │   Python 3       │
                                                                    │   flamegraph.svg │
                                                                    └──────────────────┘
```

### 数据流

```
用户点"采集" → Flask POST /api/v1/tasks
  → app.py 写 SQLite (status=PENDING)
  → Agent 心跳 Do() → Server 捎带 TaskDesc
  → Agent HotmethodChannel 从队列取任务
  → PerfProfiler.Record() → fork + execvp perf record
  → NotifyResult() 上报结果 → Server 记录 DONE
  → Analyzer: perf script → 折叠栈 → flamegraph.pl → SVG
  → 前端轮询 → <object> 嵌入 SVG
```

### 通信协议

| 链路 | 协议 | 方向 |
|------|------|------|
| Web UI ↔ API Server | HTTP/JSON | 双向 |
| API Server ↔ drop_server | gRPC (Control) | API → Server |
| Agent ↔ drop_server | gRPC (HealthCheck, Hotmethod, Init) | Agent 主动 |

## 2. 状态机迁移图

```
                    ┌──────────────────────────────┐
                    │                              │
   POST /api/v1/tasks                              │
        │                                          │
        ▼                                          │
   ┌─────────┐   Agent 心跳领取    ┌─────────┐     │
   │ PENDING  │ ─────────────────→ │ RUNNING  │     │
   └─────────┘    (捎带下发)        └─────────┘     │
        │                              │           │
        │ Agent 离线/超时              │ Agent 完成 │
        ▼                              ▼           │
   ┌─────────┐                   ┌─────────┐       │
   │ FAILED  │                   │  DONE   │       │
   └─────────┘                   └─────────┘       │
```

### 状态迁移规则

| 迁移 | 触发条件 | reason 字段 |
|------|---------|------------|
| PENDING → RUNNING | Agent 心跳捎带领取任务 | "Agent 已领取" |
| RUNNING → DONE | Agent NotifyResult 成功 | "采集完成" |
| RUNNING → FAILED | perf 执行失败/PID 不存在/超时 | 具体错误信息 |
| PENDING → FAILED | Agent 离线超过 30s | "Agent 离线" |

### 审计日志

每次状态迁移写入 `audit_log` 表（task_id + event + detail + timestamp），Web 任务详情页实时可见。

## 3. 关键决策

### 决策 1：为什么 Agent 和 Server 用 C++17？

**理由**：性能关键路径需要确定性延迟控制。
- 心跳线程每 5 秒一次，延迟抖动必须 < 1ms
- fork/exec 子进程生命周期管理需要精确的信号控制（setpgid + killpg）
- Python GIL 会导致心跳延迟抖动
- C++ RAII 保证锁、文件描述符、子进程的自动回收

**代价**：开发效率低于 Python/Go，编译时间长

### 决策 2：为什么用 gRPC 而不是 HTTP/REST？

**理由**：
- Proto 文件是类型安全的接口契约，编译器检查参数类型
- HTTP/2 多路复用，单 TCP 连接承载心跳 + 任务下发 + 结果上报
- 自动生成 C++/Python 代码，不需要手写序列化

**代价**：学习曲线陡峭（大一新生需要额外 1 天学习 proto + gRPC 概念）

### 决策 3：为什么心跳响应捎带任务（piggyback）？

**理由**：
- Agent 通常在 NAT/防火墙后面，Server 无法主动连接
- 捎带下发省一次 RPC 往返（Agent 不需要先问"有任务吗"再问"取任务"）
- 任务下发与心跳耦合，不存在"Server 有任务但 Agent 还没取"的中间状态

**代价**：心跳响应体积变大（但 TaskDesc < 1KB，可忽略）

### 决策 4：为什么 API Server 用 Python Flask？

**理由**：HTTP CRUD 不是性能瓶颈；7 天时间内 Python Flask 比 Go Gin 更快完成，把精力集中在 C++ 核心。

## 4. 取舍说明

### 做得好的

1. **采集器策略模式（IProfiler）**：新增 eBPF/pprof 采集器只需实现接口，不改现有代码——开闭原则
2. **线程安全设计**：心跳线程与工作线程完全解耦，TryPop 非阻塞保证心跳不受采集阻塞
3. **子进程生命周期管理**：fork + setpgid + execvp + 超时 killpg，拒绝 system() 防注入
4. **Proto 先行**：5 个 proto 文件定义了完整接口契约后，C++ Agent/Server/Python API Server 并行开发

### 妥协的

1. **存储层简化**：用 SQLite 替代 PostgreSQL，单文件数据库便于评审部署，但生产环境需要 PG
2. **前端简化**：纯 HTML + 内联 JS/CSS，无 React 框架，火焰图用 `<object>` 嵌入 SVG（不可交互缩放）
3. **COS 未接**：采集结果文件存本地 /tmp，生产环境需要对接 MinIO/S3 对象存储
4. **单机部署**：Server 和 Agent 打包在同一 docker-compose，生产环境 Agent 需独立部署到各台机器

## 5. AI 协作章节

### AI 帮了什么

| 场景 | AI 的作用 | 效果 |
|------|----------|------|
| Proto 文件理解 | 逐行解释 5 个 proto 的 service/rpc/message，画调用关系图 | 30 分钟内理清接口契约 |
| gRPC Echo 入门 | 提供完整 server.cpp + client.cpp 模板 | 1 小时内跑通第一个 RPC |
| TaskQueue 实现 | 解释 mutex + condition_variable 的工作原理 | 从零理解线程安全队列 |
| CMakeLists.txt | 提供 proto 编译 + 多目标构建模板 | 跳过 CMake 语法学习 |
| 每日 git commit | 提供符合规范的 commit message 模板 | 提交历史完整可读 |

### AI 哪里不行

1. **不提醒保存文件**：nano 编辑后需要手动 Ctrl+O 保存，AI 不会主动提示——导致多次文件为空
2. **WSL 路径问题**：AI 工具运行在 Windows 环境，但代码需要在 WSL 里跑，AI 无法感知当前终端是 bash 还是 WSL
3. **环境故障无法自动修复**：WSL 虚拟磁盘丢失、sudo 权限缺失等问题需要手动排查
4. **编译错误链路不透明**：AI 直接给修复方案，不解释 g++ 错误信息的含义——不利于长期学习

### 怎么改进 Prompts

1. 对 AI 说"每次给代码后提醒我保存"——避免空文件
2. 对 AI 说"遇到编译错误时先解释错误信息再给修复方案"——加深理解
3. 把 PLAN.md 喂给 AI 作为上下文——确保 AI 严格按计划执行

## 6. 性能自证

| 指标 | 测量值 | 测量方法 |
|------|--------|---------|
| 心跳处理延迟 | < 1ms | C++ mutex lock_guard，无系统调用 |
| perf 子进程启动 | < 30ms | fork + execvp，瓶颈在内核进程创建 |
| 火焰图生成 | < 5s (10MB perf.data) | perf script → 折叠栈 → flamegraph.pl |
| TaskQueue TryPop | < 1μs | mutex + deque pop_front，无阻塞 |
| SQLite 写延迟 | < 1ms | WAL 模式，单行 INSERT |
| API 响应时间 | < 10ms | Flask 本地 SQLite，无网络开销 |

## 7. 如果再给 7 天

### eBPF 采集器（优先级最高）

- 用 libbpf-go 实现 IO 延迟探针（`blk_start_request` / `blk_update_request`）
- Web 上展示 IO 延迟分布热力图
- 验收：用 `dd`/`fio` 制造 IO 抖动，eBPF 探针能采到并在 Web 上看到变化

### 智能归因（加分项）

- 把火焰图 TopN 热点 + 进程元数据 + 历史 baseline 结构化喂给 LLM
- LLM 只能调用预定义工具（查符号表、查 perf 统计、查系统配置）
- 产出可验证的归因结论（落到具体函数/行号）
- 写评测报告：造 5 个场景，统计 LLM 建议的准确率

### React 前端

- 用 D3/ECharts 做可交互火焰图（点击放大、搜索高亮、差分对比）
- 替换当前纯 HTML + `<object>` 方案

### 分布式

- 多 Agent 同时采集 + 统一聚合分析
- Server 横向扩展（gRPC 负载均衡）
